To-Do List — UUID y Autosave de Orden de Servicio

1. Identidad de la Orden de Servicio (UUID)

  • Definir formalmente el ordenServicioUuid como:

    • Identidad temporal, frontend-only

    • Distinta del _id de backend

    • Generada una sola vez al iniciar una nueva OS

  • Establecer reglas de nacimiento del UUID:

    • Se genera:

      • Al iniciar flujo “Nueva Orden de Servicio”

      • O al restaurar un borrador sin UUID

    • Nunca se regenera por:

      • re-render

      • reload

      • navegación interna

  • Persistir el ordenServicioUuid en:

    • autosave (localStorage)

    • snapshot inicial

    • payload intermedio (wizard → ingreso)

2. Namespacing y claves de LocalStorage

  • Definir formato definitivo de key: 

    css
    formIngreso:{ordenServicioUuid}:vX

  • Eliminar uso de keys globales o genéricas (formIngresoAutosave_v3)

  • Versionar el schema del autosave (vX) para futuras migraciones

  • Garantizar aislamiento:

    • una OS = una key

    • sin colisiones entre sesiones pasadas

3. Persistencia del UUID durante el flujo

  • Propagar el UUID a través de:

    • Wizard

    • IngresoPage

    • FormIngreso

    • Autosave

    • Diff

    • Snapshot

  • Asegurar que:

    • El UUID no se derive del cliente

    • El UUID no se derive del equipo

    • El UUID no se derive del payload backend

4. Snapshot y Diff (estado sucio)

  • Confirmar que:

    • El snapshot inicial incluye UUIDs

    • lineasServicio.uid se preserva intacto

  • Definir explícitamente:

    • Qué campos participan del diff

    • Qué campos se ignoran (metadata, backend-only)

  • Validar que:

    • El diff funciona de forma estable gracias a UUID

    • No hay falsos positivos por regeneración de identidad

5. Autosave

  • Asegurar que el autosave:

    • Usa siempre el ordenServicioUuid

    • No sobrescribe autosaves de otras OS

    • Guarda solo data serializable y estable

  • Confirmar que:

    • El autosave no genera nuevos UUIDs

    • Solo persiste los existentes

6. Muerte del UUID (ciclo de vida)

Casos donde el UUID muere

  • Guardado exitoso de la OS

    • Backend responde success

    • Se limpia localStorage de esa OS

    • Se invalida el UUID

  • Botón Cancelar explícito

    • Confirmación del usuario

    • Limpieza de autosave

    • UUID descartado

Casos donde el UUID no muere

  • Navegación interna

  • Refresh

  • Back del navegador

  • Cierre accidental de pestaña

  • Error de backend

7. UX: navegación y protección de datos

Estado sucio

  • Definir “estado sucio” como:

    • diff ≠ vacío

    • o autosave ≠ snapshot

Navegación con datos no confirmados

  • Implementar protección para:

    • botón atrás del navegador

    • mouse back

    • refresh

    • cerrar pestaña

Alertas

  • Implementar beforeunload como:

    • última barrera

    • mensaje genérico

    • solo si hay estado sucio

  • Interceptar navegación interna y mostrar modal propio con opciones:

    • “Seguir editando”

    • “Salir y conservar borrador”

    • “Salir y descartar”

8. Botones de la UI

  • Botón Guardar

    • Persiste en backend

    • Mata UUID

    • Limpia autosave

  • Botón Cancelar

    • Pide confirmación

    • Mata UUID

    • Limpia autosave

  • Botón Atrás del navegador

    • Nunca mata UUID automáticamente

    • Puede disparar confirmación si hay cambios

9. Restauración de borradores

  • Definir flujo:

    • detectar autosaves existentes

    • mostrar lista o restaurar último

    • rehidratar con mismo UUID

  • Validar que restaurar:

    • no genera UUID nuevo

    • no rompe diff

    • no mezcla OS distintas

10. Documentación interna (muy recomendado)

  • Documentar:

    • ciclo de vida del UUID

    • reglas de persistencia

    • reglas de muerte

    • decisiones UX

  • Dejar comentarios explícitos:

    • por qué UUID ≠ backend ID

    • por qué no se mata al usar “atrás”

11. Checklist de validación final

  • Abrir dos OS simultáneas → no colisionan

  • Refrescar → estado se conserva

  • Ir atrás → no se pierde nada sin confirmar

  • Cancelar → todo se limpia

  • Guardar → UUID desaparece

  • Autosaves antiguos no afectan OS nuevas

Cierre

Con esta lista:

  • Tienes identidad estable

  • Autosave confiable

  • UX respetuosa

  • Cero colisiones

  • Escalable a edición, duplicado, drafts, etc.

Cuando quieras, el siguiente paso natural es ordenar esto en fases de implementación o definir tests mentales / casos borde para validar que nada se escape.
